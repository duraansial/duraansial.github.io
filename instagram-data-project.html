                <h2 class="text-3xl font-semibold text-gray-900 mb-4">Project Overview</h2>

                <p class="text-gray-600 leading-relaxed">

                <h2 class="text-3xl font-semibold text-gray-900 mb-4">The Challenge</h2>

                <p class="text-gray-600 leading-relaxed">

                    When creating databases, it is often necessary to create multiple overlapping tables and a centralized system focusing on user input/output. From here, it is also important to be able to analyze trends and use a data-driven approach to understand user behavior, whether that is for external usage such as campaigns or internal usage such as user volumes & bots. Methods that forgo this will often lead to poor data quality, duplication of data, and misinterpretation of results.

                </p>

            </div>

                <h2 class="text-3xl font-semibold text-gray-900 mb-4">Technical Details & Database Design</h2>

                <p class="text-gray-600 leading-relaxed mb-4">

                    This project was built using PG Admin 4, PostgreSQL, and DBDiagram.io.

                <h3 class="text-2xl font-semibold text-gray-900 mb-4">Database Schema</h3>

                <p class="text-gray-600 leading-relaxed mb-4">

                    You can view the full database schema diagram here: <a href="https://dbdiagram.io/d/Instagram-Database-Project-Duraan-Sial-688a8981cca18e685c891409" target="_blank" class="text-orange-600 hover:underline">Instagram Database Schema Diagram</a>

                <h3 class="text-2xl font-semibold text-gray-900 mb-4 mt-8">Key Database Components:</h3>

                <ul class="list-disc list-inside text-gray-600 mb-4">

                    <li><strong>Users:</strong> stores information about users, simplified for sake of project.</li>

                    <li><strong>Tags:</strong> stores information on tags.</li>

                    <li><strong>Follows:</strong> relates user IDs to each other via follower/followee relationship.</li>

                    <li><strong>Photos:</strong> stores information on photos and relates it to users.</li>

                    <li><strong>Comments:</strong> stores comment info and relates it to users and photos.</li>

                    <li><strong>Likes:</strong> relates user IDs and Photos via ‘likes’ relationship.</li>

                    <li><strong>Photo_tags:</strong> relates photos to tags.</li>

                </ul>


                <h3 class="text-2xl font-semibold text-gray-900 mb-4 mt-8">Illustrative SQL Snippets:</h3>

                <h4 class="text-xl font-semibold text-gray-900 mb-2">Creating Tables:</h4>


                <p class="text-gray-600 mb-2">Creating Users Table</p>

                <pre><code class="language-sql">create table users(

    id serial primary key,

    username varchar(255) unique not null,

    created_at timestamp default now()

);</code></pre>


                <p class="text-gray-600 mb-2 mt-4">Creating Photos Table</p>

                <pre><code class="language-sql">create table photos(

    id serial primary key,

    image_url varchar(255) not null,

    user_id integer not null,

    created_at timestamp default now(),

    foreign key(user_id) references users(id)

);</code></pre>


                <p class="text-gray-600 mb-2 mt-4">Creating Comments Table</p>

                <pre><code class="language-sql">create table comments(

    id serial primary key,

    comment_text varchar(255) not null,

    user_id int not null,

    photo_id int not null,

    created_at timestamp default now(),

    foreign key (user_id) references users(id),

    foreign key (photo_id) references photos(id)

);</code></pre>


                <p class="text-gray-600 mb-2 mt-4">Creating Likes Table</p>

                <pre><code class="language-sql">create table likes(

    user_id int not null,

    photo_id int not null,

    created_at timestamp default now(),

    foreign key (user_id) references users(id),

    foreign key (photo_id) references photos(id),

    primary key (user_id, photo_id)

);</code></pre>


                <p class="text-gray-600 mb-2 mt-4">Creating Followers/Followees Table</p>

                <pre><code class="language-sql">create table follows(

    follower_id int not null,

    followee_id int not null,

    created_at timestamp default now(),

    foreign key (follower_id) references users(id),

    foreign key (followee_id) references users(id), -- Corrected: should reference users(id) for followee_id

    primary key (follower_id, followee_id)

);</code></pre>


                <p class="text-gray-600 mb-2 mt-4">Creating Tags Table</p>

                <pre><code class="language-sql">create table tags(

    id serial primary key not null,

    tag_name varchar(255) unique,

    created_at timestamp default now()

);</code></pre>


                <p class="text-gray-600 mb-2 mt-4">Creating Photo Tags Table</p>

                <pre><code class="language-sql">create table photo_tags(

    photo_id int not null,

    tag_id int not null,

    foreign key (photo_id) references photos(id),

    foreign key (tag_id) references tags(id),

    Primary Key (photo_id, tag_id) -- prevents users from putting the same tag on a photo

);</code></pre>

                <h2 class="text-3xl font-semibold text-gray-900 mb-4">Business Insights & Data-Driven Solutions</h2>


                <div class="mb-8">

                    <h3 class="text-2xl font-semibold text-gray-900 mb-2">Scenario 1:</h3>

                    <p class="text-gray-600 leading-relaxed mb-2">

                        We want to reward our users who have been around the longest. Who are the 5 oldest users?

                    </p>

                    <p class="text-gray-600 font-medium mb-1">SQL Query:</p>

                    <pre><code class="language-sql">select * from users order by created_at ASC limit 5;</code></pre>

                </div>


                <div class="mb-8">

                    <h3 class="text-2xl font-semibold text-gray-900 mb-2">Scenario 2:</h3>

                    <p class="text-gray-600 leading-relaxed mb-2">

                        We need to figure out when to schedule an ad campaign. What day of the week do most user register on?

                    </p>

                    <p class="text-gray-600 font-medium mb-1">Approach:</p>

                    <p class="text-gray-600 leading-relaxed mb-2">

                        Need to use created_by date and count. group by day and order by that count desc.

                    </p>

                    <p class="text-gray-600 font-medium mb-1">SQL Query:</p>

                    <pre><code class="language-sql">select to_char(created_at, 'Day') as day, count(*) as total from users group by day order by total desc;</code></pre>

                </div>


                <div class="mb-8">

                    <h3 class="text-2xl font-semibold text-gray-900 mb-2">Scenario 3:</h3>

                    <p class="text-gray-600 leading-relaxed mb-2">

                        We want to target our inactive users with an email campaign. Find the users who have never posted a photo.

                    </p>

                    <p class="text-gray-600 font-medium mb-1">Approach:</p>

                    <p class="text-gray-600 leading-relaxed mb-2">

                        We want to find users who do not have any photo_id. We will need to join tables here.

                    </p>

                    <p class="text-gray-600 font-medium mb-1">SQL Query:</p>

                    <pre><code class="language-sql">select username, photos.id as photo_posted from users left join photos on photos.user_id = users.id where photos.id is null;</code></pre>

                </div>


                <div class="mb-8">

                    <h3 class="text-2xl font-semibold text-gray-900 mb-2">Scenario 4:</h3>

                    <p class="text-gray-600 leading-relaxed mb-2">

                        We’re running a new contest to see who can get the most likes on a single photo. Who won?

                    </p>

                    <p class="text-gray-600 font-medium mb-1">SQL Query:</p>

                    <pre><code class="language-sql">SELECT users.username, photos.id, photos.image_url, COUNT(*) AS total

FROM photos

INNER JOIN likes ON likes.photo_id = photos.id

INNER JOIN users ON photos.user_id = users.id

GROUP BY photos.id, username, photos.image_url

ORDER BY total DESC

LIMIT 2;</code></pre>

                </div>


                <div class="mb-8">

                    <h3 class="text-2xl font-semibold text-gray-900 mb-2">Scenario 5:</h3>

                    <p class="text-gray-600 leading-relaxed mb-2">

                        How many times does the average user post?

                    </p>

                    <p class="text-gray-600 font-medium mb-1">Approach:</p>

                    <p class="text-gray-600 leading-relaxed mb-2">

                        Total # photos posted divided by total number of users.

                    </p>

                    <p class="text-gray-600 font-medium mb-1">SQL Query:</p>

                    <pre><code class="language-sql">select (select count(*) from photos)::numeric / (select count(*) from users)::numeric;</code></pre>

                    <p class="text-gray-600 leading-relaxed mt-2">

                        Therefore the average user posts 2 photos.

                    </p>

                </div>


                <div class="mb-8">

                    <h3 class="text-2xl font-semibold text-gray-900 mb-2">Scenario 6:</h3>

                    <p class="text-gray-600 leading-relaxed mb-2">

                        A brand wants to know which hashtags to use in a post. What are the top 5 most commonly used hashtags?

                    </p>

                    <p class="text-gray-600 font-medium mb-1">Approach:</p>

                    <p class="text-gray-600 leading-relaxed mb-2">

                        Get tags, get tag usage related to photos, then get # time a tag is used per photo, then do a join to connect them.

                    </p>

                    <p class="text-600 font-medium mb-1">SQL Query:</p>

                    <pre><code class="language-sql">select tag_name, count(tag_id) as total_usage

from photo_tags

join tags on tags.id = photo_tags.tag_id

group by tag_name

order by total_usage desc

limit 5;</code></pre>

                </div>


                <div class="mb-8">

                    <h3 class="text-2xl font-semibold text-gray-900 mb-2">Scenario 7:</h3>

                    <p class="text-gray-600 leading-relaxed mb-2">

                        We have a small problem with bots on our site… Find users who have liked every single photo on the site.

                    </p>

                    <p class="text-gray-600 font-medium mb-1">Approach:</p>

                    <p class="text-gray-600 leading-relaxed mb-2">

                        Find users, then find likes, relate users to likes, find users w/max amount of likes, we can not limit.

                    </p>

                    <p class="text-gray-600 font-medium mb-1">SQL Query:</p>

                    <pre><code class="language-sql">SELECT username, COUNT(*) as num_likes

FROM users

INNER JOIN likes ON users.id = likes.user_id

GROUP BY username

HAVING COUNT(*) = (SELECT COUNT(*) FROM photos);</code></pre>

                </div>

            </div>


            <div class="mb-10">

                <h2 class="text-3xl font-semibold text-gray-900 mb-4">Implementation & Challenges</h2>

                <p class="text-gray-600 leading-relaxed">

                    The development process involved outlining what parameters made sense in an Instagram environment and how they would interlink. I iteratively designed the necessary tables starting with the common denominator, the User. From there, I built outwards to create photos, tags, likes, followers, etc.

                </p>

                <p class="text-gray-600 leading-relaxed mt-4">

                    During the project, I encountered challenges such as creating a likes schema that would prevent one user from liking the same photo more than once (thereby creating duplicated data). To overcome this, I created a Primary Key that depended on the User’s ID and the Photo’s ID Foreign Keys.

                </p>

                <p class="text-gray-600 leading-relaxed mt-4">

                    Another challenge was figuring out how to create a tagging system that could scale with users and not en masse duplicate tags. My solution involved creating a Photo Tags Table that relates Photo info to Tag info. Then creating a PK constraint dependent on both tables. This was similar to the Likes Table.

                </p>
